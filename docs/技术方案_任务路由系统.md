# 任务路由智能系统 — 技术方案

> **目标**：旅长说任意自然语言 → 系统路由到正确的牵头负责角色，并给出依据
> **知识源**：《作战指挥手册》OCR 文本（后续可扩展更多文档）
> **架构**：双路检索（向量 + BM25）+ LLM 在线判断

---

## 全局数据流

```
                     Step 1                          Step 2                         Step 3
                 离线 · 文本处理                    离线 · 索引构建                 在线 · 路由服务
                                                                               
 ┌──────────┐    ┌──────────────┐    ┌──────────┐    ┌──────────┐    ┌──────┐    ┌──────────────┐
 │ OCR 原文  │───→│ 纠错→解析→分块 │───→│chunks.json│───→│ 灌入索引  │───→│ 双路  │───→│ LLM 路由判定  │
 │ .txt     │    │              │    │tree.json │    │          │    │ 检索  │    │ LLM 路由判定  │
 └──────────┘    │  [人工审核点] │    │roles.json│    │ Milvus   │    │ + RRF │    └──────────────┘
                 └──────────────┘    │review.json│   │ ES       │    └──────┘          │
                                     └──────────┘    └──────────┘                      ▼
                                                                               {"role": "侦察情报要素",
                                                                                "reasoning": "..."}
```

# Step 1：文本处理与结构化分块（离线）

## 输入

| 文件 | 说明 |
|------|------|
| `merged_rec_texts_cleaned_restored.txt` | OCR 扫描文本，约 35,000 字 |

## 输出

| 文件 | 说明 | 供谁使用 |
|------|------|---------|
| `corrected_text.txt` | OCR 纠错后的清洁文本 | 存档 |
| `correction_log.json` | 纠错日志（原文→修改，逐条记录） | **→ 人工审核点 ①** |
| `chapter_tree.json` | 文档结构树（章→节→项的层级关系） | Step 1 分块 + Step 3 结构查询 |
| `chunks.json` | 结构化 Chunk 数组 | **核心产出** → Step 2 |
| `role_registry.json` | 角色列表（从文本中自动发现的所有角色） | **→ 人工审核点 ②** + Step 3 |
| `review_items.json` | 需人工确认的条目（OCR 乱码段 + 角色提取不确定项） | **→ 人工审核点 ③** |

### 1.1 OCR 纠错

将文本按自然段分组（每组 800~1200 字），送 DeepSeek 纠错。

**Prompt**：

```
你是军事文献校对专家。以下是 OCR 扫描得到的军事教材段落，可能存在识别错误。
请纠正明显的 OCR 错误（错字、漏字、乱码），保持原文语义和表述不变。

规则：
1. 只修正明显的 OCR 识别错误，不改写原文表述
2. 无法推断的乱码段落，用 [OCR_ERROR] 标记
3. 输出纠正后的完整段落，再用 JSON 列出所有修改

原始段落：
{paragraph_text}
```

**产出**：
- `corrected_text.txt`：纠正后全文
- `correction_log.json`：每处修改的记录

```json
[
  {"line": 97, "original": "组织通信电D的海华盛的国任设公子防御", "corrected": "[OCR_ERROR]", "type": "garbled"},
  {"line": 124, "original": "对空全", "corrected": "对空安全", "type": "missing_char"}
]
```

**🔍 人工审核点 ①**：审核 `correction_log.json`，重点关注：
- `[OCR_ERROR]` 标记的段落：判断是否影响语义，手动修复或标记跳过
- 抽查 10% 的自动纠正是否合理

### 1.2 结构层级解析

用正则识别标题，构建文档结构树。

**层级体系**：

```
Level 1：第X章      （第二章 合同战术基本理论）
Level 2：第X节      （第一节 进攻战斗）
Level 3：X、        （一、战斗准备）
Level 4：（X）      （（一）传达任务，计划安排工作）
Level 5：X.         （1.下达预先号令）
Level 6：（X）数字  （（1）领会上级意图）
Level 7：①②③      （①本级任务）
```

**标题识别正则**：

```python
LEVEL_PATTERNS = [
    (1, r'^第[一二三四五六七八九十百]+章\s*(.+)'),
    (2, r'^第[一二三四五六七八九十百]+节\s*(.+)'),
    (3, r'^[一二三四五六七八九十]+、\s*(.+)'),
    (4, r'^（[一二三四五六七八九十]+）\s*(.+)'),
    (5, r'^(\d+)[\.．、]\s*(.+)'),
    (6, r'^（(\d+)）\s*(.+)'),
    (7, r'^[①②③④⑤⑥⑦⑧⑨⑩]\s*(.+)'),
]
```

**产出 `chapter_tree.json`**（示例片段）：

```json
{
  "title": "作战指挥手册",
  "children": [
    {
      "level": 1, "id": "ch2", "title": "第二章 合同战术基本理论",
      "children": [
        {
          "level": 2, "id": "ch2_s1", "title": "第一节 进攻战斗",
          "children": [
            {
              "level": 3, "id": "ch2_s1_p1", "title": "一、战斗准备",
              "children": [
                {
                  "level": 4, "id": "ch2_s1_p1_t1",
                  "title": "（一）传达任务，计划安排工作",
                  "children": [
                    {"level": 5, "id": "ch2_s1_p1_t1_01", "title": "1.下达预先号令"},
                    {"level": 5, "id": "ch2_s1_p1_t1_02", "title": "2.理解任务"}
                  ]
                },
                {"level": 4, "id": "ch2_s1_p1_t2", "title": "（二）组织侦察情报"}
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

**这棵树的四个用途**：
1. 驱动分块逻辑（判断叶子节点）
2. 生成 title_chain（检索时提供层级上下文）
3. 建立 parent/children 关系（结构导航）
4. 推导 context_tags（从祖先节点得出 phase/battle_type/scope）

---

### 1.3 通用分块

**分块原则**：树的叶子节点 = 一个 Chunk，非叶子节点拆出概述 Chunk + 子 Chunk。

```
规则 1（叶节点即 Chunk）：
  没有子标题的最低层级标题 + 其正文 = 一个 Chunk

规则 2（带子项的标题）：
  该标题 + 开头总述段落（子标题之前的文字）= 一个概述 Chunk（type=overview）
  每个子标题递归应用规则 1/2

规则 3（长度控制）：
  最小 50 字 → 过短的合并到父级
  最大 800 字 → 超长的按自然段拆分，共享 title_chain，用 _partN 后缀区分

规则 4（title_chain 继承）：
  每个 Chunk 携带从根到当前层级的完整标题路径
```

**分块示例**：

```
原文：
（二）组织侦察情报
  [总述段落 300字]
  1.确定情报需求。[正文 80字]
  2.概算战术目标。[正文 100字]
  3.制定侦察计划。[正文 120字]
  4.组织侦察协同。[正文 80字]

产出 5 个 Chunk：
  Chunk A（overview）："（二）组织侦察情报" + 总述 300字
  Chunk B（detail）  ："1.确定情报需求" + 正文 80字
  Chunk C（detail）  ："2.概算战术目标" + 正文 100字
  Chunk D（detail）  ："3.制定侦察计划" + 正文 120字
  Chunk E（detail）  ："4.组织侦察协同" + 正文 80字
```

### 1.4 角色提及标注

对每个 Chunk，扫描文本中**提到了哪些角色**（只标注"提到了谁"，不判断"谁负责"）。

**角色关键词库**：

```python
ROLE_KEYWORDS = [
    "指挥员", "参谋长", "筹划决策要素", "侦察情报要素",
    "指挥控制要素", "指挥保障要素", "政治工作要素",
    "后装保障要素", "后方指挥员", "作战部门", "作训部门",
    "保障部长", "机要部门", "旅政治工作部", "指挥机关",
    "战勤部门", "各业务部门", "各指挥要素",
]
```

**标注逻辑**：遍历 `ROLE_KEYWORDS`，匹配 Chunk 文本，记录出现的角色列表。

> **为什么不在这一步判断"谁负责"？**
> 因为一个 Chunk 中可能涉及多个角色、多种分工（牵头/参与/审批/配合），
> 而且"谁负责"取决于旅长具体问的是什么。
> 这个判断留给 Step 3 的 LLM 在线完成，更准确。

每个 Chunk 的完整 JSON 结构：

```json

  "chunk_type": "detail",
  "title": "3.制定侦察计划",

  "parent_id": "ch2_s1_p1_t2",
  "children_ids": [],
  "level": 5,
  "text": "制定侦察计划是对使用侦察力量的具体规划，包括编组侦察力量、明确配置位置、区分侦察任务及规定完成侦察任务的时限、申请支援的事项等。在受领任务后或指挥员定下决心后，需制定侦察计划。通常由侦察情报要素负责拟制侦察计划，报参谋长审批。",

    "phase": "战斗准备",

  },
  "roles_mentioned": ["指挥员", "侦察情报要素", "参谋长"],

    "source_file": "merged_rec_texts_cleaned_restored.txt",

```

**字段说明**：

| 字段 | 说明 | 谁用 |
|------|------|------|
| chunk_id | 唯一标识 | 全流程 |
| chunk_type | `overview`（概述）/ `detail`（具体任务）/ `theory`（理论阐述） | 检索过滤 |
| title | 当前层级标题 | 展示 |
| title_chain | 根→当前的完整层级路径 | 向量化前缀 + 展示 |
| parent_id | 父 Chunk ID | 结构导航：向上追溯 |
| children_ids | 子 Chunk ID 列表 | 结构导航：向下展开 |
| level | 层级深度（1~7） | 检索过滤 |
| text | 正文内容 | 检索 + LLM 输入 |
| context_tags | phase / battle_type / scope | 检索过滤 |
| roles_mentioned | 文本中提到的角色列表 | 检索加权 |
| metadata | 来源文件、行号、字数、是否有 OCR 错误 | 溯源 |

### 1.7 人工审核接口汇总

| 审核点 | 文件 | 审核内容 | 操作方式 |
|--------|------|---------|---------|
| **① OCR 纠错审核** | `correction_log.json` | 自动纠正是否合理；[OCR_ERROR] 段落是否可手动修复 | 编辑 `corrected_text.txt` 后重跑 1.2~1.5 |
| **② 角色列表审核** | `role_registry.json` | 角色是否完整、是否有误识别 | 直接编辑 JSON，增删角色 |
| **③ 异常条目审核** | `review_items.json` | OCR 乱码段落；roles_mentioned 为空但应有角色的 Chunk | 编辑 `corrected_text.txt` 或手动补标注 |

**设计原则**：所有需人工干预的点都输出为独立 JSON 文件，人工修改后重跑后续步骤即可，不需要改代码。

### 2.1 向量索引（Milvus）

**向量化逻辑**：

```python
# title_chain 作为前缀，让同一小节的 Chunk 在语义空间中更聚集
input_text = f"{chunk['title_chain']}\n{chunk['text']}"
embedding = bge_m3.encode(input_text)  # 1024 维
```

**Collection Schema**：

| 字段 | 类型 | 用途 |
|------|------|------|
| chunk_id | VARCHAR(PK) | 主键 |
| vector | FLOAT_VECTOR(1024) | BGE-M3 稠密向量 |
| phase | VARCHAR | 过滤条件 |
| battle_type | VARCHAR | 过滤条件 |
| scope | VARCHAR | 过滤条件 |
| chunk_type | VARCHAR | 过滤条件 |

索引类型：HNSW（数据量 <1000，HNSW 即可）

---

### 2.2 BM25 索引（Elasticsearch）

**Index Mapping**：

```json
{
  "mappings": {
    "properties": {
      "chunk_id":      {"type": "keyword"},
      "title":         {"type": "text", "analyzer": "ik_smart"},
      "title_chain":   {"type": "text", "analyzer": "ik_smart"},
      "text":          {"type": "text", "analyzer": "military_cn_analyzer"},
      "phase":         {"type": "keyword"},
      "battle_type":   {"type": "keyword"},
      "scope":         {"type": "keyword"},
      "chunk_type":    {"type": "keyword"},
      "roles_mentioned": {"type": "keyword"}
    }
  },
  "settings": {
    "analysis": {
      "filter": {
        "military_synonyms": {
          "type": "synonym",
          "synonyms": [
            "三情, 敌情 我情 战场环境",
            "后装, 后勤保障 装备保障",
            "筹划决策要素, 筹划要素",
            "侦察情报要素, 情报要素",
            "指挥保障要素, 通信保障要素"
          ]
        }
      },
      "analyzer": {
        "military_cn_analyzer": {
          "tokenizer": "ik_max_word",
          "filter": ["lowercase", "military_synonyms"]
        }
      }
    }
  }
}
```

**关键点**：
- 必须安装 IK 分词插件（ES 默认分词对中文无效）
- `roles_mentioned` 字段类型为 `keyword`，支持精确过滤
- 同义词表存在独立文件 `synonyms.txt`，新文档接入时追加

### 2.3 写入流程

```python
def build_indexes(chunks_path: str):
    """
    一次性将 chunks.json 灌入 Milvus 和 ES。

    Args:
        chunks_path: Step 1 产出的 chunks.json 路径
    """
    chunks = load_json(chunks_path)

    # 向量化
    texts = [f"{c['title_chain']}\n{c['text']}" for c in chunks]
    embeddings = bge_m3.encode(texts, batch_size=32)

    # 写入 Milvus
    milvus_data = [
        {
            "chunk_id": c["chunk_id"],
            "vector": emb.tolist(),
            "phase": c["context_tags"]["phase"],
            "battle_type": c["context_tags"]["battle_type"],
            "scope": c["context_tags"]["scope"],
            "chunk_type": c["chunk_type"],
        }
        for c, emb in zip(chunks, embeddings)
    ]
    collection.insert(milvus_data)

    # 写入 ES
    es_actions = [
        {
            "_id": c["chunk_id"],
            "_source": {
                "chunk_id": c["chunk_id"],
                "title": c["title"],
                "title_chain": c["title_chain"],
                "text": c["text"],
                "phase": c["context_tags"]["phase"],
                "battle_type": c["context_tags"]["battle_type"],
                "scope": c["context_tags"]["scope"],
                "chunk_type": c["chunk_type"],
                "roles_mentioned": c["roles_mentioned"],
            },
        }
        for c in chunks
    ]
    helpers.bulk(es_client, es_actions, index="military_manual")
```

**新文档接入**：跑 Step 1 → 增量 append 到 chunks.json → 增量写入两个索引，无需重建。

# Step 3：在线路由与检索服务

## 输入

| 来源 | 说明 |
|------|------|
| 旅长自然语言输入 | 如 "侦察计划该谁去做？" |
| Milvus + ES 索引 | Step 2 构建 |
| `chunks.json` | Chunk 原文（LLM 生成时需要） |
| `role_registry.json` | 角色列表（LLM prompt 中的候选项） |
| `chapter_tree.json` | 文档结构（结构查询） |

## 输出

| 场景 | 输出内容 |
|------|---------|
| **路由** | 牵头角色 + 参与角色 + 审批角色 + 判断依据 + 原文出处 |

---

### 3.1 整体 Pipeline

```
旅长输入
    │
    ▼
[1. 意图识别]          DeepSeek API
    │                  输出改写后的检索 query
    │
    ▼
[2. 双路并行检索]
    ├── 向量检索 Milvus Top-10
    └── BM25 检索 ES Top-10
         │
         ▼
[3. RRF 融合] → Top-5 Chunk
         │
         ▼
[4. 路由判定]
    LLM 判断谁负责
         │
         ▼
   路由结果 JSON
```

---

### 3.3 双路并行检索 + RRF 融合

```python
async def hybrid_search(query: str, filters: dict = None, top_k: int = 10):
    """
    向量 + BM25 并行检索，RRF 融合。

    Args:
        query: 意图识别提取的 search_query
        filters: 可选过滤条件，如 {"phase": "战斗准备"}
        top_k: 每路返回条数

    Returns:
        融合后的 Top-5 Chunk（含原文）
    """
    query_vector = bge_m3.encode(query)

    vector_task = milvus_search(query_vector, top_k=top_k, filters=filters)
    bm25_task = es_search(query, top_k=top_k, filters=filters)

    vector_results, bm25_results = await asyncio.gather(vector_task, bm25_task)

    fused = rrf_fuse([vector_results, bm25_results], k=60)

    # 补全 Chunk 原文（从 chunks.json 内存缓存中取）
    top_chunks = enrich_with_text(fused[:5])
    return top_chunks
```

### 3.4 路由判定

**核心变化**：不依赖预计算的 primary_owner，由 LLM **现场判断**。

```
Prompt：

你是军事指挥体制专家。旅长下达了以下指示，请根据手册规定判断相关职责分工。

## 旅长指示
{user_input}

## 检索到的手册规定
{top_5_chunks_formatted}
（格式：[来源: {title_chain}]  {text}）

## 候选角色列表
{role_registry_formatted}

请基于手册规定，输出 JSON：
{
  "lead": "牵头负责的角色（从候选列表中选）",
  "participants": ["参与协助的角色"],
  "approver": "审批角色（如有）",
  "reasoning": "2~3句判断依据，引用手册原文",
  "basis_chunk_id": "最相关的 Chunk ID",
  "confidence": 0.0~1.0
}

如果手册中没有相关规定，输出：
{"lead": "未匹配", "participants": [], "approver": null,
 "reasoning": "手册中未找到与此任务对应的职责规定", "confidence": 0}
```

**为什么路由判定全走 LLM**：
- 一个 Chunk 可能涉及多角色多分工，正则无法处理
- "谁负责"取决于旅长具体问的是什么
- LLM 能返回结构化的 lead/participants/approver，比单一 primary_owner 更有用
