# 智能体知识服务 — 技术方案

> **目标**：为多智能体指挥系统提供知识检索能力，让每个 Agent 在感知、研判、决策、执行各层级
> 都能从《作战指挥手册》中获取结构化指导（谁负责、怎么做、找谁协作、产出什么）
>
> **核心变化**：引入**流程图谱（Process Graph）**，建模"任务→角色→流程→产物"的关系网络

---

## 为什么需要知识图谱

先说清楚：单纯的 RAG（向量+BM25）能做什么、不能做什么。

| Agent 需求 | RAG 能做 | RAG 做不好 | 知识图谱补什么 |
|-----------|---------|-----------|--------------|
| "侦察计划怎么做？" | ✅ 检索到相关 Chunk | | |
| "组织侦察情报有哪些子步骤？" | ⚠️ 可能只返回部分 | 不保证返回**全部**子步骤 | 图谱直接遍历 children，**完整返回** |
| "我是参谋长，这件事我需要拉谁？" | ⚠️ 需要 LLM 从多个 Chunk 中推理 | 如果相关 Chunk 分散，可能漏人 | 图谱查 Task→Role 关系，**一次性返回所有关联角色** |
| "制定战斗方案之前需要先完成什么？" | ❌ 无法回答前置依赖 | 文本中隐含的顺序关系无法检索 | 图谱查 DEPENDS_ON 关系，**返回前置任务链** |
| "战斗准备阶段，侦察情报要素需要完成哪些任务？" | ❌ 需要同时过滤角色+阶段 | 检索不可能穷举 | 图谱查 Role→Task + Phase 过滤，**完整枚举** |

**结论**：RAG 负责"语义理解和模糊匹配"，知识图谱负责"结构化关系和完整遍历"。两者互补。

## 输入 → 输出

```
输入：OCR 扫描文本
输出：corrected_text.txt, chapter_tree.json, chunks.json, role_registry.json, review_items.json
```

> Step 1 与之前方案一致，不再重复全部细节，仅列关键点。

### 1.1 OCR 纠错

LLM 分段纠错，乱码标 `[OCR_ERROR]`。

产出：`corrected_text.txt` + `correction_log.json`

**🔍 人工审核点 ①**：审核 correction_log.json

### 1.2 结构层级解析

正则识别 7 级标题，构建文档结构树。

产出：`chapter_tree.json`

### 1.3 通用分块

叶节点即 Chunk，非叶节点拆出概述 Chunk + 子 Chunk。长度控制 50~800 字。

### 1.4 角色提及标注

扫描每个 Chunk 中出现的角色关键词，写入 `roles_mentioned` 字段。

### 1.5 角色列表生成

汇总所有角色，产出 `role_registry.json`。

**🔍 人工审核点 ②**：审核角色列表完整性

# Step 2：流程图谱构建（离线）

## 输入 → 输出

```
输入：chunks.json, chapter_tree.json, role_registry.json
输出：Neo4j 图数据库（Task/Role/Phase/Product 节点 + 关系边）
```

### 2.1 为什么叫"流程图谱"而不是"知识图谱"

FM 3-09 项目的知识图谱建模的是**概念实体**（Concept、Organization、System 等）之间的关系。

本项目建模的是**工作流程**：谁做什么事、按什么步骤、产出什么、依赖什么。所以叫流程图谱更准确。

### 2.2 Schema 设计

**节点类型**：

| 节点类型 | 标签 | 说明 | 属性 | 来源 | 预估数量 |
|---------|------|------|------|------|---------|
| 任务 | `Task` | 手册中描述的一项具体工作 | name, description, chunk_id, level | chunk_type=detail 的 Chunk | 120~150 |
| 任务组 | `TaskGroup` | 包含子任务的上级任务 | name, chunk_id, level | chunk_type=overview 的 Chunk | 30~50 |
| 角色 | `Role` | 组织中的职责角色 | name, role_id | role_registry.json | 15~20 |
| 阶段 | `Phase` | 作战阶段 | name | context_tags.phase 去重 | 4~6 |
| 战斗类型 | `BattleType` | 进攻/防御等 | name | context_tags.battle_type 去重 | 3~4 |
| 产物 | `Product` | 任务的预期产出物 | name, description | LLM 从 Chunk 文本中提取 | 30~50 |

**关系类型**：

| 关系 | 起点 | 终点 | 说明 | 来源 |
|------|------|------|------|------|
| `HAS_SUBTASK` | TaskGroup | Task/TaskGroup | 父任务包含子任务 | parent_id/children_ids |
| `NEXT_STEP` | Task | Task | 同级任务的顺序关系 | 同一 parent 下的 children 顺序 |
| `DEPENDS_ON` | Task | Task | 显式依赖（需先完成A才能做B） | LLM 从文本中提取 |
| `INVOLVES` | Task | Role | 该任务涉及此角色 | roles_mentioned |
| `LED_BY` | Task | Role | 该任务由此角色牵头 | LLM 从文本中提取 |
| `APPROVED_BY` | Task | Role | 该任务需此角色审批 | LLM 从文本中提取 |
| `PRODUCES` | Task | Product | 该任务产出此成果 | LLM 从文本中提取 |
| `BELONGS_TO` | Task | Phase | 该任务属于此阶段 | context_tags.phase |
| `IN_BATTLE_TYPE` | Task | BattleType | 该任务属于此战斗类型 | context_tags.battle_type |

### 2.3 构建流程

#### 阶段一：规则抽取（零误差骨架）

直接从 chunks.json 和 chapter_tree.json 中提取，不依赖 LLM：

```
1. 创建 Role 节点     ← role_registry.json
2. 创建 Phase 节点    ← context_tags.phase 去重
3. 创建 BattleType 节点 ← context_tags.battle_type 去重
4. 创建 Task/TaskGroup 节点 ← 每个 Chunk 对应一个节点
5. 创建 HAS_SUBTASK 关系 ← parent_id / children_ids
6. 创建 NEXT_STEP 关系  ← 同一 parent 下 children 的顺序
7. 创建 INVOLVES 关系   ← roles_mentioned
8. 创建 BELONGS_TO 关系 ← context_tags.phase
9. 创建 IN_BATTLE_TYPE 关系 ← context_tags.battle_type
```

这一步完全确定性，不会出错。

#### 阶段二：LLM 抽取（补充语义关系）

对每个 Task 节点的 Chunk 文本，调用 LLM 提取三类信息：

```
Prompt：

你是军事组织体制分析专家。以下是《作战指挥手册》中关于某项任务的描述。
请提取以下信息：

1. led_by：这项任务由谁牵头负责？（从候选角色列表中选）
2. approved_by：这项任务需要谁审批？（如有）
3. produces：这项任务应该产出什么成果/文书？（如有）
4. depends_on：这项任务在执行前需要先完成哪些前置任务？（如有，用任务名称描述）

候选角色列表：{role_list}

任务标题：{title}
上下文路径：{title_chain}
任务描述：{chunk_text}

输出严格 JSON：
{
  "led_by": "角色名" | null,
  "approved_by": "角色名" | null,
  "produces": ["产物名1", "产物名2"] | [],
  "depends_on": ["前置任务描述1"] | []
}
```

**抽取结果处理**：
- `led_by` → 创建 `LED_BY` 关系
- `approved_by` → 创建 `APPROVED_BY` 关系
- `produces` → 创建 `Product` 节点 + `PRODUCES` 关系
- `depends_on` → 模糊匹配已有 Task 节点名称，创建 `DEPENDS_ON` 关系；匹配不上的标记 review

**🔍 人工审核点 ③**：审核 LLM 抽取的 LED_BY、DEPENDS_ON 关系是否准确

### 2.5 Neo4j 索引

```cypher
CREATE INDEX task_name FOR (t:Task) ON (t.name);
CREATE INDEX task_chunk FOR (t:Task) ON (t.chunk_id);
CREATE INDEX role_name FOR (r:Role) ON (r.name);
CREATE INDEX product_name FOR (p:Product) ON (p.name);

CREATE FULLTEXT INDEX task_fulltext
FOR (t:Task|TaskGroup) ON EACH [t.name, t.description];
```

# Step 3：索引构建（离线）

## 输入 → 输出

```
输入：chunks.json（Step 1）, Neo4j 图谱（Step 2）
输出：Milvus 向量索引 + ES BM25 索引
```

### 3.1 向量索引（Milvus）

```python
input_text = f"{chunk['title_chain']}\n{chunk['text']}"
embedding = bge_m3.encode(input_text)  # 1024 维
```

Collection 字段：chunk_id(PK), vector, phase, battle_type, scope, chunk_type

### 3.2 BM25 索引（Elasticsearch）

ES Mapping 字段：chunk_id, title, title_chain, text, phase, battle_type, scope, chunk_type, roles_mentioned

使用 IK 分词 + 军事术语同义词表。

### 3.3 验收

| # | 检查项 | 预期 |
|---|--------|------|
| 1 | 三个存储文档数一致 | Milvus = ES = chunks.json 条目数 |
| 2 | 图谱节点数 | ≈ Task 数 + Role 数 + Product 数 |
| 3 | 向量检索 | "谁来做侦察计划" → Top-5 含相关 Chunk |
| 4 | BM25 检索 | "筹划决策要素" → 精确命中 |
| 5 | 图谱查询 | 查 Task "制定侦察计划" 的 LED_BY → 返回"侦察情报要素" |

# Step 4：在线知识服务（Agent 调用接口）

Step 4 分两个部分：

- **Part A — 知识服务侧**：薄接口，纯数据查询，**不含 LLM**
- **Part B — Agent 侧编排指南**：Agent 代码如何组合调用这些接口，以及**哪里用 LLM**

---

## Part A：知识服务 API（服务端）

### A.1 设计原则

| 原则 | 说明 |
|------|------|
| **薄接口** | 每个接口只做一件事：查图谱 / 搜向量 / 搜 BM25 / 读 Chunk |
| **无 LLM** | 知识服务内部不调用任何 LLM，所有 LLM 调用由 Agent 侧负责 |
| **结构化响应** | 全部返回 JSON，Agent 程序直接解析 |
| **双模式入参** | 图谱接口同时支持 `chunk_id`（精确）和 `task_name`（模糊匹配） |

### A.2 接口总览

| # | 接口 | 方法 | 用途 | 数据源 | LLM |
|---|------|------|------|--------|-----|
| ① | `/api/v1/resolve` | POST | 自然语言 → 匹配到的 Task 节点 | 向量 + BM25 + 图谱 | ❌ |
| ② | `/api/v1/graph/role_tasks` | GET | 查某角色在某阶段的所有任务 | Neo4j | ❌ |
| ③ | `/api/v1/graph/task_roles` | GET | 查某任务涉及的所有角色 | Neo4j | ❌ |
| ④ | `/api/v1/graph/task_decompose` | GET | 查某任务的子步骤 | Neo4j | ❌ |
| ⑤ | `/api/v1/graph/task_prerequisites` | GET | 查某任务的前置依赖 | Neo4j | ❌ |
| ⑥ | `/api/v1/graph/task_products` | GET | 查某角色/阶段应产出什么 | Neo4j | ❌ |
| ⑦ | `/api/v1/graph/task_detail` | GET | 查某任务的完整信息（图谱+原文） | Neo4j + Chunk | ❌ |
| ⑧ | `/api/v1/search` | POST | 语义检索，返回相关 Chunk | 向量 + BM25 | ❌ |
| ⑨ | `/api/v1/chunk/{chunk_id}` | GET | 读取单个 Chunk 原文 | Chunk 存储 | ❌ |

> **所有接口均无 LLM。** LLM 的参与点在 Agent 侧。

### A.3 接口详细定义

#### ① 任务解析（自然语言 → Task 节点）

**这是最关键的桥梁接口**：Agent 手里只有自然语言，需要先"定位"到图谱中的 Task 节点。

```
POST /api/v1/resolve

Request:
{
  "query": "组织侦察情报",
  "top_k": 3
}

内部流程（无 LLM）：
1. query → BGE-M3 编码 → Milvus 向量搜索 Top-10
2. query → ES BM25 搜索 Top-10
3. RRF 融合 → Top-K 候选 Chunk
4. 用 chunk_id 关联 Neo4j Task 节点，补充图谱元数据

Response:
{
  "matches": [
    {
      "chunk_id": "ch2_s1_p1_t2",
      "task_name": "（二）组织侦察情报",
      "title_chain": "第二章 > 第一节 > 一、战斗准备 > （二）组织侦察情报",
      "node_type": "TaskGroup",
      "phase": "战斗准备",
      "has_subtasks": true,
      "score": 0.96
    },
    {
      "chunk_id": "ch2_s1_p1_t2_03",
      "task_name": "3.制定侦察计划",
      "title_chain": "第二章 > 第一节 > 一、战斗准备 > （二）组织侦察情报 > 3.制定侦察计划",
      "node_type": "Task",
      "phase": "战斗准备",
      "has_subtasks": false,
      "score": 0.82
    }
  ]
}
```

**设计要点**：
- 不只返回 chunk_id，还返回 `node_type`（Task/TaskGroup）和 `has_subtasks`，让 Agent 代码知道下一步该调 `task_decompose` 还是 `task_detail`
- 无需 LLM——向量+BM25+RRF 足以完成语义匹配

#### ② 角色职责查询

```
GET /api/v1/graph/role_tasks?role=侦察情报要素&phase=战斗准备

内部：单条 Cypher 查询
MATCH (r:Role {name: $role})<-[rel:INVOLVES|LED_BY|APPROVED_BY]-(t)
WHERE (t)-[:BELONGS_TO]->(:Phase {name: $phase})
RETURN t.name, t.chunk_id, type(rel), ...

Response:
{
  "role": "侦察情报要素",
  "phase": "战斗准备",
  "tasks": [
    {
      "task_name": "确定情报需求",
      "responsibility": "involves",
      "chunk_id": "ch2_s1_p1_t2_01",
      "parent_task": "组织侦察情报"
    },
    {
      "task_name": "制定侦察计划",
      "responsibility": "led_by",
      "chunk_id": "ch2_s1_p1_t2_03",
      "parent_task": "组织侦察情报"
    }
  ]
}
```

#### ③ 任务角色分工

```
GET /api/v1/graph/task_roles?chunk_id=ch2_s1_p1_t2_03
# 也支持：?task_name=制定侦察计划（走 Neo4j 全文索引模糊匹配）

内部：Cypher 查询
MATCH (t {chunk_id: $chunk_id})-[rel:INVOLVES|LED_BY|APPROVED_BY]->(r:Role)
RETURN r.name, type(rel)

Response:
{
  "task": "制定侦察计划",
  "chunk_id": "ch2_s1_p1_t2_03",
  "roles": {
    "led_by": "侦察情报要素",
    "involves": ["指挥员", "侦察情报要素", "参谋长"],
    "approved_by": "参谋长"
  }
}
```

#### ④ 任务拆解

```
GET /api/v1/graph/task_decompose?chunk_id=ch2_s1_p1_t2

内部：Cypher 查询 HAS_SUBTASK + NEXT_STEP 排序 + 每个子任务的 LED_BY/PRODUCES
MATCH (tg {chunk_id: $chunk_id})-[:HAS_SUBTASK]->(sub)
OPTIONAL MATCH (sub)-[:LED_BY]->(r:Role)
OPTIONAL MATCH (sub)-[:PRODUCES]->(p:Product)
OPTIONAL MATCH (sub)-[:DEPENDS_ON]->(dep)
RETURN sub, r, p, dep ORDER BY sub.level, sub.name

Response:
{
  "task": "组织侦察情报",
  "chunk_id": "ch2_s1_p1_t2",
  "subtasks": [
    {
```

## Part B：Agent 侧编排指南

### B.1 总体原则

```
Agent 代码（主干逻辑 + LLM）  ←→  知识服务（薄接口，纯数据）
         ^                                  ^
         │                                  │
    LLM 在这一侧                     无 LLM，纯查询
    负责"理解"和"决策"               负责"检索"和"遍历"
```

**LLM 参与的三种场景**（均在 Agent 代码中）：
1. **意图理解**：旅长说了一句话，LLM 判断这是什么意图 → 决定调哪些接口
2. **信息综合**：拿到多个接口的结构化结果后，LLM 综合推理出行动方案
3. **文本生成**：需要对外输出自然语言（报告、回复）时，LLM 生成

### B.2 Agent 各层级的调用编排

#### 研判层：这件事跟我有关吗？

```
触发：Agent 收到一条消息（旅长指令 / 其他 Agent 的@提及）
目标：判断自己是否是相关方

Agent 代码流程：
┌──────────────────────────────────────────────────────────────────────┐
│ 1. [代码] 调 resolve(query=消息文本, top_k=3)                        │
│    → 得到匹配的 Task 列表                                            │
│                                                                      │
│ 2. [代码] 对每个匹配的 Task，调 task_roles(chunk_id=...)             │
│    → 得到每个 Task 涉及的角色                                        │
│                                                                      │
│ 3. [代码] 检查自己的角色名是否出现在任何一个 Task 的                  │
│    led_by / involves / approved_by 中                                │
│    → if 命中: 标记相关，记录 responsibility 类型                     │
│    → if 未命中：忽略 / 标记为无关                                    │
│                                                                      │
│ ⚡ 是否用 LLM：通常不需要。角色名精确匹配即可。                      │
│    例外：如果旅长说了模糊的话（"帮我查查敌情"），resolve 返回的       │
│    匹配置信度不高，可选调 LLM 做二次判断。                           │
└──────────────────────────────────────────────────────────────────────┘
```

#### 决策层（场景 A）：任务拆解 + 人员匹配

```
触发：研判层判定"这件事跟我有关且需要牵头"
目标：拆解任务为子步骤，确定每步由谁负责

Agent 代码流程：
┌──────────────────────────────────────────────────────────────────────┐
│ 1. [代码] 调 resolve(query=旅长指令) → matched_task                  │
│                                                                      │
│ 2. [代码] 判断 matched_task.has_subtasks                             │
│    → if true: 调 task_decompose(chunk_id)                            │
│      → 得到子步骤列表 + 每步的 led_by + depends_on                   │
│    → if false: 说明是叶子任务，无需拆解                              │
│                                                                      │
│ 3. [代码] 对缺少 led_by 的子步骤，调 task_roles(chunk_id)           │
│    → 补全角色分工                                                    │
│                                                                      │
│ 4. ⭐ [LLM] 将以上结构化数据 + 旅长原始指令 灌入 Prompt：           │
│    """                                                                │
│    旅长指令：{command}                                                │
│    知识库检索结果：                                                   │
│    - 任务拆解：{decompose_result}                                    │
│    - 各步骤角色分工：{roles_per_step}                                │
│                                                                      │
│    请基于以上信息，输出任务执行计划：                                 │
│    1. 确认子任务列表和执行顺序                                       │
│    2. 为每个子任务分配执行人                                         │
│    3. 如果旅长指令有知识库未覆盖的内容，补充说明                     │
│    """                                                                │
│    → LLM 输出结构化执行计划                                          │
│                                                                      │
│ 5. [代码] 解析 LLM 输出 → 创建子任务 → 分发给对应 Agent             │
└──────────────────────────────────────────────────────────────────────┘
```

#### 决策层（场景 B）：旅长指令超出手册覆盖范围

```
触发：resolve 返回的匹配分数很低（如 < 0.6），说明手册中没有直接对应的任务

Agent 代码流程：
┌──────────────────────────────────────────────────────────────────────┐
│ 1. [代码] 调 resolve(query=旅长指令) → score < 0.6                   │
│                                                                      │
│ 2. [代码] 退而求其次，调 search(query=旅长指令, top_k=5)             │
│    → 拿到语义最相关的 Chunk（可能不是精确任务，但有参考价值）        │
│                                                                      │
│ 3. ⭐ [LLM] 将检索结果 + 旅长指令 灌入 Prompt：                     │
│    """                                                                │
│    旅长指令：{command}                                                │
│    知识库中未找到精确匹配的任务，但检索到以下相关内容：              │
│    {search_chunks}                                                   │
│                                                                      │
│    请参考以上内容，推理出合理的任务拆解方案和人员分工。              │
│    如果信息不足以做出完整判断，请标明不确定的部分。                  │
│    """                                                                │
│    → LLM 基于不完整信息做"尽力而为"的决策                            │
└──────────────────────────────────────────────────────────────────────┘
```

#### 执行层：具体怎么做这件事？

```
触发：Agent 被分配了一个具体子任务
目标：了解任务的详细执行步骤和注意事项

Agent 代码流程：
┌──────────────────────────────────────────────────────────────────────┐
│ 1. [代码] 调 task_detail(chunk_id=被分配任务的chunk_id)              │
│    → 得到：原文 + 图谱关系（牵头人/审批人/产物/依赖）                │
│                                                                      │
│ 2. [代码] 如果需要更多背景，调 search(query=任务名, top_k=3)         │
│    → 补充相关但非直接关联的 Chunk                                    │
│                                                                      │
│ 3. [代码] 调 task_products(chunk_id=当前任务)                         │
│    → 知道本次任务应该产出什么                                        │
│                                                                      │
│ 4. ⭐ [LLM] 综合以上信息生成执行方案：                               │
│    """                                                                │
│    你的任务：{task_name}                                              │
│    任务描述：{full_text}                                              │
│    预期产物：{products}                                               │
│    前置条件：{prerequisites}（已完成 / 未完成）                       │
│    参考资料：{related_chunks}                                         │
│                                                                      │
│    请输出你的执行方案，包括具体步骤、需要协调的事项、产出物格式。    │
│    """                                                                │
│    → LLM 生成具体执行步骤                                            │
└──────────────────────────────────────────────────────────────────────┘
```

#### 输出层：生成报告或回复

```
触发：子任务完成，需要汇报或回复旅长
目标：生成自然语言输出

Agent 代码流程：
┌──────────────────────────────────────────────────────────────────────┐
│ 1. [代码] 调 task_products(chunk_id=当前任务)                         │
│    → 知道产出物应该是什么格式                                        │
│                                                                      │
│ 2. [代码] 调 search(query="XX报告格式要求", top_k=3)                  │
│    → 如果手册中有报告格式说明，检索出来                              │
│                                                                      │
│ 3. ⭐ [LLM] 基于执行结果 + 格式要求 → 生成报告                      │
└──────────────────────────────────────────────────────────────────────┘
```

### B.3 LLM 参与点总结

| Agent 层级 | 调用知识服务（无 LLM） | Agent 侧 LLM 参与 |
|-----------|----------------------|-------------------|
| **研判层** | `resolve` → `task_roles` | 通常不需要；低置信度时做二次判断 |
| **决策层** | `resolve` → `task_decompose` → `task_roles` | **必须**：综合结构化数据 + 旅长意图，做任务拆解和选人决策 |
| **执行层** | `task_detail` → `search` → `task_products` | **必须**：将知识转化为具体执行方案 |
| **输出层** | `task_products` → `search` | **必须**：生成自然语言报告/回复 |

### B.4 完整调用链示例（端到端）

```
旅长说："组织侦察情报准备"

=== 参谋长 Agent ===

[研判层]
  → resolve("组织侦察情报准备") → chunk_id: ch2_s1_p1_t2, score: 0.96
  → task_roles(ch2_s1_p1_t2) → involves: [参谋长, 侦察情报要素, ...]
  → 判定：我(参谋长)在 involves 列表中 ✅ 且无 led_by → 我来协调

[决策层]
  → task_decompose(ch2_s1_p1_t2) → 4 个子步骤
  → task_roles(ch2_s1_p1_t2_01) → involves: [指挥员]
  → task_roles(ch2_s1_p1_t2_02) → involves: [侦察情报要素]
  → task_roles(ch2_s1_p1_t2_03) → led_by: 侦察情报要素, approved_by: 参谋长
  → task_roles(ch2_s1_p1_t2_04) → involves: [侦察情报要素, 参谋长]
  → ⭐ LLM: 综合以上，输出执行计划：
     子任务1 "确定情报需求" → 分配给指挥员
     子任务2 "概算战术目标" → 分配给侦察情报要素
     子任务3 "制定侦察计划" → 分配给侦察情报要素（我审批）
     子任务4 "组织侦察协同" → 分配给侦察情报要素+我
  → [代码] 创建 4 个子任务，@侦察情报要素 和 @指挥员

=== 侦察情报要素 Agent ===

[研判层]
  → 收到参谋长分配的子任务 "制定侦察计划"，chunk_id 已知
  → 我是 led_by → 我来做 ✅

[执行层]
  → task_detail(ch2_s1_p1_t2_03) → 得到完整描述 + 产物 + 依赖
  → task_prerequisites(ch2_s1_p1_t2_03) → 需先完成"概算战术目标"
  → [代码] 检查"概算战术目标"状态 → 已完成 ✅
  → search("侦察计划编制要点") → 补充参考
  → ⭐ LLM: 生成侦察计划草案

[输出层]
  → ⭐ LLM: 将草案格式化 → 发给参谋长审批
```
